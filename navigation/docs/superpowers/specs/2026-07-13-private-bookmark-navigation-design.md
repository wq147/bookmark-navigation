# 私人书签导航站设计规范

## 1. 目标

把现有数百个多层级浏览器书签建设成一个可公网访问、需要账号密码登录的私人导航站。服务器上的书签数据库是唯一主数据源；Chrome、Edge、Firefox 等浏览器通过手动导入、导出标准 Netscape Bookmark HTML 与导航站交换数据，不依赖浏览器账号同步或扩展。

第一版重点解决大量书签的查找、分类、维护和安全导入导出问题，不把系统做成 NAS 图标面板或网页存档服务。

## 2. 已确认的约束

- 单用户使用，公网访问必须登录。
- 使用 Docker Compose 部署。
- HTTPS 由云服务器现有 Nginx 或 Caddy 反向代理提供。
- 接受手动导出浏览器 HTML、上传导航站，以及从导航站下载 HTML 后导回浏览器。
- 保留现有多层目录、连续编号和 `bookmark_policy.json` 分类体系。
- 复用 `bookmark_organizer.py`、`bookmark_numbering.py` 和 `bookmark_audit.py` 中已经验证的解析、URL 规范化、分类、去重和审计逻辑。

## 3. 第一版范围

### 3.1 包含

- 单用户登录、退出和会话管理。
- 按标题、URL、域名、目录和备注进行全局搜索。
- 多层目录树浏览、展开、折叠和计数。
- 书签新增、编辑、移动、删除和批量操作。
- 目录新增、重命名、移动、删除和重新编号。
- 浏览器书签 HTML 导入、差异预览和确认应用。
- 标准 Netscape Bookmark HTML 全量导出。
- JSON 数据备份导出。
- 导入和高风险批量操作前自动备份。
- 操作记录和从操作前备份恢复。
- 桌面端与移动端自适应界面。

### 3.2 不包含

- 浏览器扩展或自动同步。
- 多用户、分享、协作和公开目录。
- 标签体系。
- 网页快照、全文存档和阅读批注。
- 自动链接健康检查。
- 根据导入文件缺少某条书签而自动删除服务器书签。

## 4. 页面设计

### 4.1 桌面端

采用三栏工作台：

1. 顶部工具栏包含站点入口、全局搜索、新增书签、导入导出和账户菜单。
2. 左栏为可折叠多层目录树，显示目录书签数量。
3. 中栏为当前目录或搜索结果，默认使用适合大量数据的紧凑列表，并允许切换卡片视图。
4. 右栏为选中书签的快速编辑面板，包括标题、URL、目录和备注。

首页展示常用书签、最近添加和最近修改。搜索框是页面最突出的交互入口，搜索结果应支持键盘选择和直接打开。

### 4.2 移动端

采用单栏布局。搜索位于顶部，目录树放入侧边抽屉，书签详情使用独立编辑页或底部面板。第一版不要求移动端拖动排序，但必须支持完整的增删改和导入导出入口。

## 5. 系统架构

系统采用单体应用，降低部署和维护成本：

```text
浏览器
  -> HTTPS Nginx/Caddy
  -> 书签导航应用容器
       |- Vue 3 前端
       |- FastAPI API
       |- 书签分类与 HTML 转换模块
       `- SQLite
  -> 宿主机持久化目录
       |- 数据库
       |- 自动备份
       `- 应用日志
```

- 前端使用 Vue 3 和 Vite，构建产物由同一应用提供。
- 后端使用 FastAPI，提供认证、书签、目录、搜索、导入导出、备份和操作记录 API。
- SQLite 作为主数据库；当前数据规模无需引入独立数据库服务。
- Docker 镜像使用多阶段构建，运行时保持为单个应用容器。
- 数据库、备份和必要日志通过 Docker volume 或宿主机目录持久化。

## 6. 数据模型

### 6.1 用户

保存用户名、Argon2 密码哈希、启用状态、密码更新时间和最后登录时间。第一版只允许一个正常用户，不提供公开注册。

### 6.2 目录

保存目录 ID、父目录 ID、不带编号前缀的基础名称、同级顺序、创建时间和修改时间。父目录引用形成任意深度的树；界面显示和 HTML 导出时根据顺序生成编号前缀。一级目录沿用策略文件中的固定名称，其他同级目录必须从 `01_` 开始连续且不重复。重新编号只改变顺序或显示结果，不改变目录 ID。

### 6.3 书签

保存书签 ID、标题、原始 URL、规范化 URL、目录 ID、备注、浏览器原始属性、创建时间和修改时间。规范化 URL 用于查重，原始 URL 用于访问和无损导出。

### 6.4 操作与备份

操作记录保存操作类型、摘要、影响数量、时间和关联备份。备份保存 SQLite 快照或完整逻辑快照的路径、校验信息、创建原因和创建时间。

## 7. 导入与合并流程

1. 用户上传 Netscape Bookmark HTML。
2. 后端只解析并生成临时导入批次，不立即改动主数据。
3. 使用现有 URL 规范化规则比较服务器数据，生成新增、重复、字段冲突、建议移动和无法分类统计。
4. 新 URL 依据 `bookmark_policy.json` 分类；无法可靠判断的书签进入 `00_待整理`。
5. URL 已存在时，默认保留服务器标题、目录和备注。预览中展示差异，用户可逐项或批量选择覆盖字段。
6. 导入文件中缺少服务器已有 URL 不表示删除，不生成自动删除操作。
7. 用户确认前创建自动备份，再在单个数据库事务中应用变更。
8. 应用后运行目录编号和审计；若事务或审计失败则回滚并保留错误报告。

重复判断基于现有 `normalize_url` 规则，包括协议与端口规范化、尾部斜杠处理和常见跟踪参数移除。导入预览必须显示规范化导致的合并，避免用户误判。

## 8. 导出流程

- HTML 导出包含完整目录树、连续编号、书签标题、原始 URL、时间戳、图标和能够保留的浏览器属性。
- 导出前运行只读审计；存在阻断性结构异常时不生成正式导出，并返回明确错误。
- Chrome 等浏览器会把导入内容追加到新目录，因此界面需提示用户先清理旧导入目录或避免重复导入。
- JSON 导出用于完整备份，包含 HTML 难以表达的备注、操作元数据和稳定 ID。

## 9. 安全设计

- 密码使用 Argon2 哈希，不保存明文密码。
- 登录后使用带 `HttpOnly`、`Secure`、`SameSite=Lax` 属性的会话 Cookie。
- 登录接口实施按 IP 和用户名的失败限速，并记录失败事件。
- 所有数据和导入导出接口都要求认证；未认证访问只显示登录页。
- 修改接口实施 CSRF 防护，反向代理必须正确转发协议、主机和客户端 IP。
- 上传只接受限制大小的 HTML 文件；文件名不参与服务器路径拼接。
- 导入解析不得发起对书签 URL 的网络请求。
- 应用容器使用非 root 用户运行，持久化目录采用最小必要权限。

## 10. 错误处理与恢复

- 导入解析错误返回文件级错误和尽可能准确的位置，不修改主数据。
- 批量修改使用事务，部分失败不得留下半完成状态。
- 删除目录前必须选择移动其内容或连同内容删除；后者需要二次确认并创建备份。
- 自动备份采用保留策略，默认保留最近 30 个操作备份和每日备份 30 天；具体数量可通过环境变量调整。
- 恢复操作本身也先创建备份，保证可以撤销一次错误恢复。

## 11. 测试与验收

### 11.1 自动测试

- 单元测试覆盖 HTML 解析/生成、URL 规范化、重复判断、目录编号、分类规则和冲突决策。
- API 测试覆盖登录权限、CRUD、搜索、导入预览、确认导入、导出和恢复。
- 数据库测试覆盖事务回滚、唯一约束和目录树移动边界。
- 前端测试覆盖搜索、目录切换、编辑表单和导入预览的主要状态。

### 11.2 真实数据验收

- 导入仓库外的私人验收 fixture，应得到与源文件一致的唯一书签集合且不产生新的重复 URL。
- 当前一级目录和多层子目录结构应保持一致。
- 从导航站导出 HTML，再由解析器重新读取，书签数量、URL 集合和目录路径应一致。
- 使用另一个浏览器导出的测试文件导入时，应正确区分新增、重复和冲突，且不会自动删除服务器数据。
- 未登录用户无法访问任何书签内容或导出文件。

## 12. 部署与配置

Docker Compose 暴露一个仅供反向代理访问的应用端口，并挂载持久化数据目录。敏感配置放在不提交版本库的 `.env` 中，至少包括初始用户名、初始密码或初始化令牌、会话密钥、可信代理和外部访问 URL。

首次启动执行数据库迁移；若没有用户，则通过一次性初始化命令创建管理员。后续升级先自动备份数据库，再运行迁移。项目提供健康检查、示例反向代理配置、备份说明和升级/回滚步骤。

## 13. 成功标准

- 多个浏览器无需登录同一浏览器账号，也能通过手动 HTML 往返使用同一套主书签。
- 数百个书签能够通过搜索或目录在数秒内定位。
- 一次导入不会静默覆盖导航站中的人工修改，也不会根据缺失项误删书签。
- 数据可通过标准 HTML 离开系统，不形成平台锁定。
- 单机 Docker Compose 部署后，日常维护只需更新容器和备份持久化目录。
